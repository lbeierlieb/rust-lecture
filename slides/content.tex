\begin{frame}
    \frametitle{Compilation vs. Interpretation}
    Compilation
    \begin{itemize}
        \item Compiler translates source code to machine instructions.
        \item Potential for low overhead and high performance.
        \item The compiler can check the code before any execution happens.
        \item Examples: C, C++
    \end{itemize}
\end{frame}

\begin{frame}
    \frametitle{Compilation vs. Interpretation}
    Interpretation
    \begin{itemize}
        \item Interpreter executes source code directly.
        \pause
        \item Interpretation overheaded causes performance penalty.
        \item Interpreter is required for execution.
        \item Errors occur only at runtime.
        \pause
        \item Convenient!
        \item Examples: Python, Javascript, Bash, Ruby, Nix
    \end{itemize}
    \pause
    \vfill
    Rust is a compiled language.
\end{frame}

\begin{frame}[fragile]
    \frametitle{Static vs. Dynamic Typing}
    Dynamic Typing
    \vfill
    \begin{itemize}
        \item Great for flexibility and convenience.
        \item Can make code easier to understand (more consise).
        \pause
        \vspace{.3cm}
        \item Can make code harder to understand (missing type information).
        \pause
        \item Type checking has a performance penalty.
        \item Errors occur only at runtime.
    \end{itemize}
    \pause
    \vfill
    Runtime type errors are not great for a language with safety as a goal.
\end{frame}

\begin{frame}[fragile]
    \frametitle{Static vs. Dynamic Typing}
    Static Typing
    \pause
    \vfill
    \begin{itemize}
        \item Type checks at compile time.
        \item No performance overhead.
        \pause
        \vspace{.3cm}
        \item Type definitions introduce verbosity.
        \item Expressiveness and complexity depend on the type system of a language (C vs. Java vs. Rust vs. Haskell).
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Data Races}
    \begin{minted}[linenos]{c}
// Data-race example in C
void* worker(void* arg) {
    for (int i = 0; i < 1000000; i++) {
        counter++;
    }
    return NULL;
}
int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);
    <...> // wait for threads to finish and inspect counter
}
    \end{minted}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Data Races}
    The final value of \texttt{counter} depends on the execution order.
    \vfill
    \begin{itemize}
        \item \texttt{t1} reads \texttt{counter}: \texttt{100}.
        \item \texttt{t2} reads \texttt{counter}: \texttt{100}.
        \item \texttt{t2} writes \texttt{counter}: \texttt{101}.
        \item \texttt{t1} writes \texttt{counter}: \texttt{101}. One increment got lost.
    \end{itemize}
    \pause
    \vfill
    Rust guarantees the absense of data races.
\end{frame}

\begin{frame}[fragile]
    \frametitle{Error Handling}
    Many languages use exceptions for error cases.
    \pause
    \vfill
    \begin{minted}[linenos]{java}
// Java
String input = readStdIn();
int num = Integer.parseInt(input);
System.out.println("You entered the number " + num);
    \end{minted}
    \pause
    \vfill
    \begin{itemize}
        \item This snippet is valid Java.
        \pause
        \item \texttt{parseInt} throws a \texttt{NumberFormatException} for unparsable strings.
        \item The function signature does not indicate this fact.
        \pause
        \item Java's design implies error handling is optional.
        \item If you want handle all exceptions, how do you know which can occur?
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Error Handling}
    \begin{itemize}
        \item C does not have exceptions.
        \item A common pattern for functions is
        \begin{itemize}
            \item Return value indicates success or gives an error code.
            \item The actual return value is passed and set by reference.
        \end{itemize}
    \end{itemize}
    \pause
    \vfill
    \begin{minted}[linenos]{java}
struct timeval tv;

int err = gettimeofday(&tv, NULL);

if (err != 0) {
    printf("Error getting time\n");
    return 1;
}
    \end{minted}
    \pause
    \vfill
    Nothing prevents you from accessing \texttt{tv} without checking the return value.
\end{frame}

\begin{frame}[fragile]
    \frametitle{Error Handling}
    Rust differentiates two kinds of errors.
    \pause
    \vfill
    \begin{itemize}
        \item Unrecoverable errors: Stop the current thread of execution.
        \pause
        \item Recoverable errors
        \begin{itemize}
            \item The type system forces functions that can error to represent the error cases in their return values.
            \pause
            \item If a fallible function returns a value, you have to handle the error case to access this value.
            \item If a fallible function does not return a value, the compiler will warn if you ignore the potential error.
        \end{itemize}
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Cost of Abstractions}
Basic Java implementation to sum the squares of all even numbers in \texttt{List<Integer> numbers}:
    \begin{minted}[linenos]{java}
int sum = 0;
for (int i = 0; i < numbers.size(); i++) {
    int num = numbers[i];
    if (num % 2 == 0) {
        sum += num * num;
    }
}
    \end{minted}
    \pause
    \vfill
Java 8 introduced \texttt{Stream}s for functional-style collection processing.
    \begin{minted}[linenos]{java}
int sum = numbers.stream()
    .filter(x -> x % 2 == 0)
    .mapToInt(x -> x * 2)
    .sum();
    \end{minted}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Cost of Abstractions}
    \begin{itemize}
        \item \href{https://www.baeldung.com/java-streams-vs-loops}{java-streams-vs-loops} benchmarked streams over 3x slower than loops.
        \pause
        \vspace{.3cm}
        \item Rust aims to provide zero-cost abstractions.
        \item The following Rust code should compile to the same instructions as a loop with manual operators.
    \end{itemize}
    \pause
    \vfill
    \begin{minted}[linenos]{rust}
let sum = numbers.iter()
    .filter(|x| x.is_multiple_of(2))
    .map(|x| x.pow(2))
    .sum();
    \end{minted}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Memory Management}
    Processes are split into segments.
    \vfill
    \begin{itemize}
        \item \textit{text}: Instructions
        \item \textit{rodata}: Read-only data (e.g., string literals)
        \item \textit{data}: Initialized global variables
        \item \textit{bss}: Uninitialized global variables
        \pause
        \vspace{.3cm}
        \item \textit{stack}: Local variables, data for function calls. Managed by the instructions.
        \pause
        \vspace{.3cm}
        \item \textit{heap}: Runtime-allocated memory. There are different management strategies.
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Memory Management}
    There are different ways for manual memory management to go wrong.
    \pause
    \vfill
    \begin{itemize}
        \item Memory leak
        \item Double free
        \pause
        \item Use after free
        \item Out-of-bounds access
    \end{itemize}
    \pause
    \vfill
    Leaking memory is safe, the others must not happen in an memory-safe language.
\end{frame}

\begin{frame}[fragile]
    \frametitle{Memory Management}
    Many languages (e.g., Java) rely on garbage collection to free heap memory when no longer needed.
    \pause
    \vfill
    Different approaches for garbage collection exist.
    One example is \textit{Mark and Sweep} garbage collection.
    \begin{itemize}
        \item Mark stage: Going through all references to heap objects and marking the objects as still in use. The references can be global variables or local variables (on the stack).
        \item Sweep stage: Going through all heap objects and freeing the unmarked ones.
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Memory Management}
    Garbage collection solves a class for memory-safety problems and is convenient, but it does not come for free.
    \pause
    \vfill
    \begin{itemize}
        \item Code has to run to perform the garbage collection, causing overhead.
        \item It is not deterministic when the garbage collector will run.
        \item Programs require a runtime environment including the garbage collection.
    \end{itemize}
    \pause
    \vfill
    These properties make garbage collection less suitable for:
    \begin{itemize}
        \item Embedded devices
        \item Operating-system kernels and drivers
        \item Low-latency applications\footnote{\url{https://discord.com/blog/why-discord-is-switching-from-go-to-rust}}
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Memory Management}
    Another common approach is reference counting (e.g., used by CPython).
    \pause
    \vfill
    \begin{minted}[linenos]{python}
# Python
text = "Hello world"
print(text)
text = "something else"
    \end{minted}
    \pause
    \vfill
    \begin{itemize}
        \item Line 2: Allocation of \texttt{"Hello world"}; \texttt{text} is the only reference.
        \item Line 3: Passing a copy of the reference to \texttt{print}; reference count is two.
        \item Line 3: \texttt{print} returns and drops its reference.
        \item Line 4: \texttt{text} stores another reference; reference count of \texttt{"Hello world"} drops to zero.
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Memory Management}
    Reference counting is deterministic, but:
    \pause
    \vfill
    \begin{itemize}
        \item Reference counting causes overhead.
        \item Cyclic references can cause memory leaks.
    \pause
    \vfill
        \item Rust uses ownership-based memory management---memory safe, deterministic, no overhead.
        \item You can also find (explicit) uses of reference counting in Rust.
    \end{itemize}
\end{frame}

fn area(shape: Shape) -> f64 {

\begin{frame}[fragile]
    \frametitle{\texttt{match} Expressions}
    \begin{itemize}
        \item Each case of a \texttt{match} expression is called a \textit{match arm}.
        \item Each arm has to evaluate to the same type.
        \pause
        \vfill
        \vspace{.5cm}
        \item We have observed arms of the form \texttt{<pattern> => <expression>}.
        \item Arms can also have the form \texttt{<pattern> <guard> => <expression>}.
        \pause
        \vfill
        \vspace{.5cm}
        \item Usually, an arm is terminated by a \texttt{,}
        \item For the last arm, the \texttt{,} is optional but idiomatic.
        \item If the expression is a block, the \texttt{,} is optional. Skipping it is idiomatic.
        \pause
        \vfill
        \vspace{.5cm}
        \item Examples!
    \end{itemize}
\end{frame}

\begin{frame}[fragile]

// Arbitraty score: Two points for large triangles, one for other
// triangles, none for other shapes
fn score(shape: Shape, min_base: f64) -> u8 {
    match shape {
        Shape::Triangle { base, height: _ } if base >= min_base => 2,
        Shape::Triangle { base: _, height: _ } => 1,
        _ => 0,
    }
}
    \end{minted}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Ownership}
    \begin{itemize}
        \item In Rust, each value has exactly one \textit{owner}.
        \item When the owner goes out of scope, the value gets \textit{dropped}.
    \end{itemize}
    \pause
    \vfill
    \begin{minted}[linenos]{rust}
fn main() {
    let x = 3;  // x owns the i32 value
    {
        let mut y = 4;  // y owns another i32 value
        y += 10;
    }  // y goes out of scope, the 14 gets dropped
}  // x goes out of scope, the 3 gets dropped
    \end{minted}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Ownership}
    \begin{itemize}
        \item A type can define clean-up actions that run when one of its values gets dropped.
        \pause
        \item Examples
        \begin{itemize}
            \item A \texttt{struct} storing a pointer to heap memory: Free the memory.
            \item A \texttt{struct} storing a raw file descriptor (\texttt{i32}): Close the file.
            \item A \texttt{struct} representing that you hold a mutex lock: Release the lock.
        \end{itemize}
        \pause
        \item This mechanism is really powerful. You cannot forget to clean up, it happens automatically!
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Ownership}
    \begin{itemize}
        \item For the given examples, making bitwise copies would break the whole concept.
        \pause
        \item By default, values get \textit{moved}---not copied---when assigned to a new owner.
    \end{itemize}
    \pause
    \vfill
    \begin{minted}[linenos]{rust}
struct WrappedI32(i32);

fn main() {
    let x = WrappedI32(3);  // x is the owner
    let y = x;  // ownership moves from x to y.
                // x must no longer be accessed!
}  // y goes out of scope, the value gets dropped
    \end{minted}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Ownership}
    Move vs. Copy
    \begin{itemize}
        \item The compiler cannot know whether your \texttt{struct}s and \texttt{enum}s are safe to copy.
        \item Types have to be especially marked to be safe for copying.
        \pause
        \vspace{.3cm}
        \item Primitives types are copyable.
        \item Arrays and tuples containing only copyable types are copyable.
        \pause
        \vspace{.3cm}
        \item You can use \texttt{\#[derive(Clone, Copy)]} to make your types copyable (we will understand in detail later).
        \item If a field of a type is not copyable, the whole type cannot be copyable. 
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Ownership}
    \begin{minted}[linenos]{rust}
#[derive(Debug, Clone, Copy)]
struct WrappedI32(i32);

fn main() {
    let mut x = WrappedI32(3);
    let y = x;  // copy! x keeps its value, y gets a copy
    x.0 += 10;
    println!("x = {x:?}, y = {y:?}");
}
    \end{minted}
    \pause
    \vfill
    Output:
    \begin{minted}{text}

    \end{minted}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Borrowing/References}
    Passing ownership to everywhere where you need the value is inconvenient.
    \pause
    \begin{itemize}
        \item Returning ownership in return value is annoying, especially with multiple values.
        \item Bitwise copies become more expensive with larger types.
        \pause
        \vspace{.3cm}
        \item You can create references to values.
        \item This process is called \textit{borrowing}.
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Borrowing/References}
    \begin{minted}[linenos]{rust}
fn main() {
    let number = 3;
    let reference: &i32 = &number;

\begin{frame}[fragile]
    \frametitle{Borrowing/References}
    Summary of the borrowing rules:
    \vfill
    \begin{itemize}
        \item While a mutable reference exists, there must not be other references.
        \item While a mutable reference exists, the owner must not use the value.
        \pause
        \item Thus, a mutable reference has exclusive access to the value.
        \pause
        \item Anything else would allow for data races.
        \pause
        \vspace{.3cm}
        \item Multiple immutable references can exist simultaneously.
        \item While a immutable reference exists, the owner must not modify the value.
        \pause
        \item Shared read-only access is unproblematic.
    \end{itemize}
\end{frame}

\begin{frame}[fragile]
    \frametitle{Borrowing/References}
    Another perspective:
    \vfill
    \begin{itemize}
        \item When someone passes you an immutable reference, the compiler guarantees that the referenced value will never change. 
        \pause
        \item When someone passes you a mutable reference, the compiler guarantess that nobody else can access the value.
        \vspace{.3cm}
        \pause
        \item Great for you: Avoiding obscure non-local bugs.
        \item Great for compiler optimization.
    \end{itemize}
    \pause
    \vfill
    We look at examples of references as function arguments soon.
\end{frame}
